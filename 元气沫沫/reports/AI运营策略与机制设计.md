## 一、四个方向的总览

| 方向 | 当前状态 | 目标 | 核心机制 |
|------|----------|------|----------|
| 智能订货 | 人工经验 | 数据驱动 | 需求预测引擎 + 补货规则引擎 |

## 二、机制一：智能订货

### 2.1 为什么人工经验不够

**证据**：朝阳站三次长假（五一、六一、端午）均出现集中报损。端午单日报损¥2,251，其中奥尔良三明治110份¥1,430——备货量是节日前一天的高峰量，但节日当天客流已归零。

**根因**：人工订货依赖店长"感觉"，无法同时考虑：
- 历史同期销量（18个月数据未被利用）
- 天气因子（小时监测表已有天气数据）
- 节假日系数（五一/六一/端午数据未被结构化）
- 库存结余动态（库存监测表数据未与订货联动）

### 2.2 机制设计：三层模型

```
┌─────────────────────────────────────────────────┐
│                  智能订货引擎                      │
├─────────────────────────────────────────────────┤
│ Layer 1: 基准预测                                 │
│   输入：同品类历史N天移动平均 × 工作日/周末系数       │
│   输出：每门店每品类基准订货量                       │
├─────────────────────────────────────────────────┤
│ Layer 2: 因子修正                                 │
│   天气因子：雨天 ×0.85（工作日）/ ×0.5（周末）       │
│   温度因子：>35°C → 饮品 ×1.3                        │
│   节假日因子：节前+2天 ×1.5，节日当天 ×1.2，节后 ×0.7  │
│   特殊事件：门店活动/投诉恢复/新品上线                 │
├─────────────────────────────────────────────────┤
│ Layer 3: 约束校验                                 │
│   库存上限：订货后总量 ≤ 陈列容量 × 保质期系数        │
│   库存下限：订货后总量 ≥ 日均销量 × 1.5              │
│   报损控制：报损率 >5% 的门店自动减量20%              │
└─────────────────────────────────────────────────┘
```

### 2.3 数据管道

```
小时监测表(API) ─┐
库存监测表(API) ─┼─→ 订货引擎 ─→ 每日订货建议表 ─→ 工作群推送
经营数据总览 ────┘                     │
天气API ─────────┘              ┌──────┴──────┐
                               │ 店长确认/微调 │
                               └──────┬──────┘
                                      ↓
                               美团管家提交订货
```

### 2.4 落地步骤

| 阶段 | 时间 | 内容 | 验收标准 |
|------|------|------|----------|
| Phase 1 | Week 1 | Excel历史数据 → Python脚本读取，输出基准预测表 | 输出12门店×4品类订货量 |
| Phase 2 | Week 2 | 加入天气+节假日因子，朝阳站专项调优 | 朝阳站报损率<3% |
| Phase 3 | Week 3-4 | 脚本定时执行，输出推送至工作群 | 每日8:00前自动输出 |

## 三、机制二：盘点异常监测

### 3.1 为什么人工审核不够

**证据**：
- 当前流程：店长盘点 → 区经审核 → 异常生成指定盘点单 → 第二天复盘
- 问题：时效差（T+1才发现）、依赖区经经验、无法发现系统性异常

### 3.2 异常规则引擎

```
┌──────────────────────────────────────────────────┐
│                 异常检测规则库                      │
├────────────┬─────────────────────────────────────┤
│ 规则类型    │ 触发条件                              │
├────────────┼─────────────────────────────────────┤
│ 🔴 库存异常  │ 结余率 <10% 且 日均销量 > 历史均值      │
│            │ 结余率 >80% 且 日均销量 < 历史均值×0.3   │
├────────────┼─────────────────────────────────────┤
│ 🔴 销量异常  │ 当日销量 > 历史均值 + 3σ               │
│            │ 当日销量 < 历史均值 - 2σ 且 库存充足      │
├────────────┼─────────────────────────────────────┤
│ 🟡 跨店异常  │ 同一品类 >50%门店同天缺货               │
│            │ 同一品类 >50%门店同天结余>70%            │
├────────────┼─────────────────────────────────────┤
│ 🟡 趋势异常  │ 连续3天结余率递减 >10pp                │
│            │ 连续3天报损递增                          │
├────────────┼─────────────────────────────────────┤
│ 🟢 新品异常  │ 新品上线7天后销量<日均5                │
│            │ 新品上线30天后仍<日均10                  │
└────────────┴─────────────────────────────────────┘
```

### 3.3 实时异常看板

**基于现有小时监测表增强**：
- 现有：门店销量 + 库存结余
- 增强：每条数据行追加异常标记（🔴🟡🟢）
- 新增：异常汇总卡片（页面顶部）
- 新增：异常自动推送（飞书/微信工作群）

## 四、机制三：营销活动策略

### 4.1 当前问题

> "不是单纯的降价，让用户薅羊毛，而是能满足用户什么需求，提供什么价值"

- 当前仅有的"活动"：6/8鼓楼买三明治送三元酸奶（供应商免费赠送）

### 4.2 策略框架：从"降价"到"价值"

```
┌─────────────────────────────────────────────────┐
│               营销活动设计框架                      │
├──────────────┬──────────────┬───────────────────┤
│ 用户场景      │ 核心需求      │ 价值主张            │
├──────────────┼──────────────┼───────────────────┤
│ 早高峰通勤    │ 快 + 饱腹     │ "30秒带走早餐"       │
│              │              │ 三明治+饮品 ¥15套餐   │
├──────────────┼──────────────┼───────────────────┤
│ 午间补能      │ 解馋 + 提神   │ "下午两点加油站"      │
│              │              │ 咖啡+面包 ¥12套餐    │
├──────────────┼──────────────┼───────────────────┤
│ 晚高峰回家    │ 晚餐+明日早餐  │ "今天买明天吃"       │
│              │              │ 买三明治送吐司券(次日) │
├──────────────┼──────────────┼───────────────────┤
│ 周末出游      │ 便携 + 分享   │ "元气野餐包"         │
│              │              │ 4款组合 ¥39          │
├──────────────┼──────────────┼───────────────────┤
│ 高铁出行(朝阳站)│ 路上吃       │ "一路元气"           │
│              │              │ 三明治+水+零食 ¥20   │
└──────────────┴──────────────┴───────────────────┘
```

### 4.3 套餐设计原则

| 原则 | 说明 |
|------|------|
| **跨品类捆绑** | 高毛利(热狗汉堡58.5%) + 低毛利(饮品47.6%) → 综合毛利>50% |
| **价格锚定** | 套餐价 = 单买总价 × 0.85，让用户感觉"赚了" |
| **时段定向** | 早高峰套餐 vs 午后套餐，不同时段可选不同组合 |
| **库存联动** | 结余率>70%的品类优先进入套餐，减少报损 |

### 4.4 活动评估模板

每个活动上线后需追踪：

| 指标 | 衡量方式 | 基准 |
|------|----------|------|
| 套餐渗透率 | 套餐订单/总订单 | 目标 >15% |
| 客单价变化 | 活动门店 vs 对照门店 | 提升 >¥0.5 |
| 关联品类提升 | 套餐内品类的非套餐销量变化 | 不下降 |
| 利润影响 | 增量毛利 - 折扣成本 | 正 |

## 五、机制四：会员体系设计

### 5.1 通勤场景的独特优势

元气沫沫的用户不是"逛商场顺便买"，而是"每天必须经过"。这决定了：

> **会员体系的核心不是"拉新"，而是"锁定高频"**

一个每天通勤的用户，每年有约500次购买机会（250个工作日×早晚2次）。如果客单价¥12，年消费潜力¥6,000。

### 5.2 会员价值公式

```
会员价值 = 购买频次 × 客单价 × 留存年限 - 会员成本

如果通过会员体系：
- 购买频次从 3次/周 → 4次/周（+33%）
- 客单价从 ¥12 → ¥14（+17%，通过套餐引导）
- 留存从 1年 → 2年

则单会员终身价值从 ¥1,872 → ¥5,824（+211%）
```

### 5.3 三种会员模式对比

| 模式 | 机制 | 优势 | 劣势 | 适用 |
|------|------|------|------|------|
| **订阅制** | ¥99/月，每日任选套餐 | 锁定高频、预收款 | 需防"薅羊毛" | ⭐推荐 |
| **积分制** | ¥1=1积分，100分兑换 | 低门槛、灵活 | 留存弱 | 辅助 |
| **储值制** | 充值¥100送¥20 | 现金流好 | 对¥12客单太重 | 不推荐 |

### 5.4 推荐方案：早餐订阅卡

```
┌─────────────────────────────────────────┐
│          元气沫沫 · 早餐月卡              │
├─────────────────────────────────────────┤
│  ¥99/月                                 │
│  每日1份指定套餐（三明治+饮品 ¥15价）      │
│  实际单餐成本 ¥6.6（毛利率50%）           │
│  月成本 ¥145/人，月收入 ¥99/人            │
│                                         │
│  关键假设：                              │
│  - 用户25天中实际使用18天                 │
│  - 未使用天数缓冲了成本                   │
│  - 附加销售（加购零食等）补回利润         │
│                                         │
│  盈亏测算：                              │
│  - 18天使用 → 成本¥119 → 亏损¥20/人      │
│  - 附加销售¥3/次 → +¥54 → 盈利¥34/人     │
│  - 500个会员 → 月利润¥17,000              │
└─────────────────────────────────────────┘
```

### 5.5 实施路径

| 阶段 | 内容 |
|------|------|
| Phase 1 | 美团后台调研：会员管理功能、发券/积分能力 |
| Phase 2 | 小程序绑定会员：扫码点餐 → 自动累积 |
| Phase 3 | 朝阳站 + 平安里试点早餐月卡（最高频门店） |
| Phase 4 | 数据复盘后扩至更多门店 |


| 脚本 | 功能 | 输入 | 输出 | 频率 |

| `anomaly_check.py` | 异常检测 | 小时监测表API | 异常清单 | 每日 |

## 七、优先级矩阵

```
                    影响大
                      │
         ┌────────────┼────────────┐
         │  盘点异常   │  智能订货   │
         │  (2周)     │  (4周)     │
         │            │            │
  容易 ──┼────────────┼────────────┼── 困难
         │            │            │
         │  营销活动   │  会员体系   │
         │  (持续)    │  (2-3月)   │
         │            │            │
         └────────────┼────────────┘
                      │
                    影响小
```

**执行顺序**：智能订货（先做，数据最成熟）→ 盘点异常（在订货基础上增强）→ 营销活动（持续迭代）→ 会员体系（需美团后台支持）
